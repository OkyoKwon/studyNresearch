## 1. 배경 및 문제

### 1.1 현재 상황

핀테크, 규제 등 관심 도메인의 지식이 노션, 블로그, 메모 앱 등 여러 곳에 흩어져 있다. 구조화되지 않은 메모는 시간이 지나면 맥락을 잃고, 검색이 어렵고, 도메인 간 관계를 파악하기 힘들다.

### 1.2 문제 정의

- **사용자 문제**: 학습한 도메인 지식을 체계적으로 축적하고 빠르게 참조할 수 있는 단일 허브가 없다. 출퇴근 중 모바일로 간단히 확인하거나, 데스크톱에서 깊이 탐색하는 패턴이 현재 도구로는 지원되지 않는다.
- **비즈니스 문제**: 지식 축적이 지속되지 않으면 동일 주제를 반복 조사하게 되어 시간이 낭비된다. 구조화된 지식 자산이 쌓이지 않는다.
- **영향 범위**: 사용자 1명 (본인). 그러나 축적된 지식은 향후 공개 모드 전환 시 외부 공유 가능한 자산이 된다.

### 1.3 기회

- AI를 활용해 콘텐츠 초안 생성을 자동화하면 지식 축적 속도를 극대화할 수 있다
- 정적 사이트 + GitHub Pages 조합으로 무료 호스팅, 버전 관리, CI/CD를 동시에 확보한다
- 도메인 확장 가능한 구조를 초기에 설계하면 장기적으로 개인 지식 포트폴리오로 발전시킬 수 있다

## 2. 솔루션

### 2.1 제안 솔루션

마크다운 기반 정적 사이트 생성기를 활용하여 도메인별로 구조화된 지식 위키를 구축한다. GitHub Pages로 배포하고, AI가 콘텐츠 초안을 생성하며, 도메인 추가 템플릿을 통해 확장 가능한 구조를 제공한다.

### 4.1 핵심 화면 목록

| 화면 | 설명 |
|------|------|
| 홈 (index) | 사이트 소개, 도메인 목록 카드, 최근 업데이트 |

| 개념 문서 | 특정 개념/용어의 상세 설명 |
| 제품 카탈로그 | 대표 제품/서비스 비교표 또는 개별 제품 상세 |
| 규제/트렌드 | 규제 프레임워크, 국가별 현황 정리 |

### 4.2 UX 원칙

- **3탭 규칙**: 홈에서 어떤 문서든 3번의 클릭/탭 이내 도달
- **스캔 가능한 구조**: 제목, 소제목, 불릿을 활용하여 빠른 훑기 가능
- **일관된 문서 구조**: 모든 도메인 문서가 동일한 템플릿을 따라 학습 비용 최소화
- **모바일 우선 읽기**: 출퇴근 중 모바일 독서 경험 최적화 (긴 가로 스크롤 금지, 적절한 폰트 크기)

### 4.3 엣지 케이스 및 에러 상태

| 상태 | 처리 방안 |
|------|----------|
| 빈 도메인 (문서 없음) | "이 도메인은 아직 준비 중입니다" 안내 메시지 + 기여 방법 링크 |
| 검색 결과 없음 | "검색 결과가 없습니다. 다른 키워드로 시도해보세요" 메시지 |
| 404 페이지 | 커스텀 404 페이지: 홈으로 돌아가기 링크 + 검색 바 |
| 깨진 내부 링크 | CI 단계에서 링크 체크 (mkdocs build --strict) |
| 다이어그램 미렌더링 | Mermaid 미지원 환경에서 코드블록 원문 표시 (graceful degradation) |
| 빌드 실패 | GitHub Actions에서 빌드 실패 시 이전 배포 유지, 실패 알림 |

## 5. 기술 아키텍처

### 5.1 정적 사이트 생성기: MkDocs Material

**선택 근거:**

| 기준 | MkDocs Material | Docusaurus | Hugo |
|------|----------------|------------|------|
| 마크다운 네이티브 | O (마크다운 전용) | O (MDX) | O |
| 검색 내장 | O (lunr.js 기반, 한국어 플러그인 가능) | O | 외부 플러그인 필요 |
| 반응형 기본 제공 | O (Material Design) | O | 테마 의존 |
| 설정 복잡도 | 낮음 (YAML 설정) | 중간 (JS 설정) | 중간 |
| GitHub Pages 배포 | 내장 지원 (`mkdocs gh-deploy`) | 수동 설정 필요 | 수동 설정 필요 |
| 위키/문서 특화 | O (문서 사이트 전문) | O (문서+블로그) | 범용 |
| 다이어그램 지원 | Mermaid 플러그인 내장 | 플러그인 필요 | 외부 도구 |
| 1인 프로젝트 적합성 | 높음 (최소 설정) | 중간 | 중간 |

**결론**: MkDocs Material은 문서 사이트에 특화되어 있고, YAML 기반 설정으로 복잡도가 낮으며, 검색/반응형/다이어그램을 기본 제공한다. 1인 프로젝트에서 최소한의 설정으로 최대 기능을 활용할 수 있어 최적의 선택이다.

## 7. 초기 도메인 상세

### 7.1 PG (Payment Gateway) 서비스

**개요**: 온라인 결제를 중개하는 서비스. 가맹점과 카드사/은행 사이에서 결제 정보를 안전하게 전달하고 승인 처리를 수행한다.

**다루어야 할 핵심 문서:**

| 문서 | 핵심 내용 |
|------|----------|
| 개요 (index.md) | PG란 무엇인가, 왜 필요한가, 온라인 결제 생태계에서의 역할 |
| 핵심 개념 (concepts.md) | 매입, 발급, 정산, 인증결제 vs 비인증결제, 빌링키, VAN, 간편결제 |
| 결제 플로우 (payment-flow.md) | 온라인 결제의 전체 흐름 (고객 --> 가맹점 --> PG --> VAN --> 카드사/은행), 다이어그램 포함 |
| 제품 비교 (products/index.md) | 국내외 주요 PG 비교표 |

### 7.2 MOR (Merchant of Record) 서비스

**개요**: 판매 거래의 법적 판매자(Merchant of Record) 역할을 대행하는 서비스. PG가 결제 처리만 담당하는 것과 달리 MOR은 세금, 환불, 규제 준수까지 대행한다.

**다루어야 할 핵심 문서:**

| 문서 | 핵심 내용 |
|------|----------|
| 개요 (index.md) | MOR이란 무엇인가, 왜 등장했는가, SaaS 판매에서의 역할 |
| 핵심 개념 (concepts.md) | Merchant of Record, Reseller 모델, 세금 대행, VAT/GST, 환불 처리, 구독 관리 |
| PG vs MOR (pg-vs-mor.md) | PG와 MOR의 핵심 차이점 비교표, 언제 PG를 쓰고 언제 MOR을 쓰는가 |
| 제품 비교 (products/index.md) | 주요 MOR 서비스 비교표 |
| Paddle (products/paddle.md) | SaaS 특화, 글로벌 세금 처리, B2B/B2C 지원 |
| Lemon Squeezy (products/lemon-squeezy.md) | 인디 개발자 친화, 간결한 가격, 디지털 제품 특화 |
| FastSpring (products/fastspring.md) | 소프트웨어 판매 특화, 엔터프라이즈 지원, 다양한 결제수단 |
| 트렌드 (trends.md) | MOR 시장 성장, SaaS 글로벌 판매의 세금 복잡성 증가, MOR vs 자체 세금 처리 |

### 7.3 가상자산 규제

**개요**: 암호화폐, 토큰, 스테이블코인 등 가상자산에 대한 각국 규제 프레임워크. FATF 권고, 국가별 입법 현황, 규제 기관을 다룬다.

## 11. FAQ

| 질문 | 답변 |
|------|------|
| 왜 MkDocs Material을 선택했는가? | 문서 사이트 특화, YAML 기반 최소 설정, 검색/반응형/다이어그램 기본 제공, GitHub Pages 배포 내장 지원. 1인 프로젝트에서 설정 비용을 최소화하면서 필요한 기능을 모두 충족한다. |
| 왜 Docusaurus나 Hugo가 아닌가? | Docusaurus는 React/JS 생태계로 오버헤드가 크고, Hugo는 문서 특화 기능(검색, 네비게이션)을 별도 설정해야 한다. 위키/문서 사이트 목적에는 MkDocs Material이 가장 적합하다. |
| 인증이 정말 필요 없는가? | 현재 단계에서는 1인 사용이므로 불필요하다. 향후 외부 공유 시 GitHub Pages의 private repo 설정 또는 선택적 퍼블리싱으로 대응 가능하다. |
| AI가 작성한 콘텐츠의 정확도는? | 모든 AI 생성 콘텐츠는 사용자가 직접 검토하고 수정한 후 커밋한다. AI는 초안 생성 도구이며, 최종 품질 책임은 사용자에게 있다. |
| 도메인을 어떻게 추가하는가? | `templates/domain-template/`을 `docs/domains/{new-domain}/`으로 복사하고, `mkdocs.yml`의 nav 섹션에 추가하면 된다. 상세 절차는 도메인 추가 가이드(FR-011)에 문서화한다. |
| 검색이 한국어를 지원하는가? | MkDocs 내장 검색은 기본적으로 영문 최적화이다. 한국어 지원을 위해 `lang: ko` 설정 및 필요 시 플러그인 추가를 검토한다. 정확도가 부족할 경우 Q3에서 개선한다. |

    PG (Payment Gateway) - 핵심 개념¶

    PG 도메인을 이해하기 위한 필수 용어와 개념을 정리한다. 각 개념의 정의, 중요성, 실제 예시를 포함한다.

    < PG (Payment Gateway) 개요로 돌아가기

    용어 정리표¶

    용어 영문 한줄 정의
    매입 Acquiring 카드사가 가맹점 매출 전표를 접수하고 대금을 정산하는 과정
    발급 Issuing 카드사가 소비자에게 카드를 발급하고 관리하는 역할
    정산 Settlement PG/카드사가 가맹점에 수수료를 제외한 매출 대금을 지급하는 과정
    인증결제 Authenticated Payment 본인인증(ISP, 안심클릭 등)을 거친 후 승인하는 결제
    비인증결제 Non-authenticated Payment 별도의 본인인증 없이 카드번호·유효기간 등으로 승인하는 결제
    빌링키 Billing Key 정기결제를 위해 카드 정보 대신 사용하는 PG 발급 토큰
    VAN Value Added Network 카드 승인 전문을 카드사에 중계하는 부가가치통신망
    간편결제 Easy Payment 사전 등록된 결제수단으로 간소화된 인증만으로 결제
    PCI DSS Payment Card Industry Data Security Standard 카드 데이터 보안 국제 표준
    3D Secure 3-Domain Secure 온라인 카드 결제 시 추가 본인인증 프로토콜

    매입(Acquiring)과 발급(Issuing)¶

    매입(Acquiring)¶

    정의: 카드 결제에서 가맹점 측을 담당하는 역할이다. 매입사(Acquirer)는 가맹점과 계약을 맺고, 카드 매출 전표를 접수하며, 가맹점에 대금을 정산한다.

    왜 중요한가: PG가 결제를 중계하더라도, 최종적으로 자금을 이동시키는 것은 매입사다. 매입 구조를 이해해야 정산 주기, 수수료율, 차지백 처리 흐름을 파악할 수 있다.

    실제 예시: 한국에서는 카드사가 매입과 발급을 동시에 수행한다(겸영 구조). 예를 들어 삼성카드는 카드를 발급하면서 동시에 가맹점 매출도 매입한다. 반면 해외에서는 Visa/Mastercard 네트워크 위에 매입사(Worldpay, Adyen 등)와 발급사(Chase, Barclays 등)가 분리되어 있다.

    발급(Issuing)¶

    정의: 소비자에게 카드를 발급하고, 결제 승인·한도 관리·청구를 담당하는 역할이다.

    왜 중요한가: 결제 승인 여부는 발급사가 결정한다. 카드 한도 초과, 분실 신고, 해외결제 차단 등의 이유로 승인이 거절되면 PG를 통한 결제도 실패한다.

    실제 예시: 소비자가 신한카드를 사용해 쿠팡에서 결제하면, 신한카드가 발급사 역할을 하며 해당 거래의 승인 여부를 결정한다.

    정산(Settlement)¶

    정의: 결제가 완료된 거래에 대해 PG사 또는 카드사가 수수료를 제외한 금액을 가맹점 계좌로 입금하는 과정이다.

    왜 중요한가: 정산 주기(D+N)와 수수료율은 가맹점의 현금 흐름에 직접적인 영향을 준다. 스타트업에게 정산 주기 1일 차이가 자금 운용에 큰 영향을 줄 수 있다.

    정산 흐름:

    1. 카드사가 매출 전표를 매입 처리
    2. 카드사가 PG사에 수수료를 제외한 금액을 지급 (보통 D+2~D+3)
    3. PG사가 자체 수수료를 제외한 금액을 가맹점에 지급

    실제 예시: Toss Payments는 D+1 정산(영업일 기준 다음 날 입금)을 제공하며, 이는 자금 회전이 빠른 소규모 가맹점에 큰 장점이다. 반면 전통 PG는 D+3~D+5 정산이 일반적이다.

    정산 주기 표기법

    D+N에서 D는 결제 발생일(Day), N은 영업일 기준 소요일수를 의미한다. D+1이면 결제 다음 영업일에 정산, D+3이면 3영업일 후 정산이다.

    인증결제 vs 비인증결제¶

    인증결제 (Authenticated Payment)¶

    정의: 결제 과정에서 카드 소유자의 본인인증을 거친 후 승인하는 방식이다. 한국에서는 ISP(인터넷안전결제), 안심클릭, 간편결제 인증 등이 해당된다.

    왜 중요한가: 본인인증을 거치므로 부정결제 위험이 낮고, 차지백 시 가맹점의 책임이 줄어든다. 한국 전자금융거래법에서는 30만원 이상 결제 시 본인인증을 의무화하고 있다.

    실제 예시: 고객이 온라인 쇼핑몰에서 50만원짜리 상품을 카드 결제하면, PG사 결제창에서 카드 선택 후 카드사 앱으로 이동하여 비밀번호·생체인증으로 본인확인을 거친다.

    비인증결제 (Non-authenticated Payment)¶

    정의: 별도의 본인인증 절차 없이 카드번호, 유효기간, CVC 등 카드 정보만으로 승인하는 방식이다. 정기결제(구독), 키인결제(Key-in)가 대표적이다.

    왜 중요한가: 사용자 경험이 간편하지만, 부정결제 위험이 높아 PCI DSS 준수가 필수적이다. 정기결제 서비스(넷플릭스, SaaS 등)에서 주로 사용한다.

    실제 예시: 넷플릭스 구독 시 최초 1회만 카드를 등록하면, 이후 매월 빌링키를 이용해 자동으로 비인증결제가 이루어진다.

    빌링키(Billing Key)¶

    정의: PG사가 카드 정보를 안전하게 저장한 뒤 발급하는 카드 대체 토큰이다. 가맹점은 실제 카드번호를 보관하지 않고 빌링키만으로 반복 결제를 요청할 수 있다.

    왜 중요한가:

    • 보안: 가맹점이 카드번호를 직접 저장하지 않아 PCI DSS 부담이 줄어든다
    • 편의성: 정기결제·원클릭 결제 구현의 핵심 수단이다
    • 유효성: 카드 갱신/분실 시 빌링키가 무효화될 수 있어 관리 로직이 필요하다

    실제 예시: SaaS 서비스가 Toss Payments API로 빌링키를 발급받으면, 매월 결제일에 해당 빌링키로 POST /billing/payments 요청만 보내면 자동 결제가 된다.

    [카드번호 등록] → [PG가 빌링키 발급: billing_key_abc123] → [매월 빌링키로 결제 요청]

    VAN (Value Added Network)¶

    정의: 카드 승인 전문(전자문서)을 가맹점/PG에서 카드사로 중계하는 부가가치통신망이다. 한국 고유의 결제 인프라 계층이다.

    왜 중요한가: 한국 결제 시스템의 독특한 구조를 이해하는 핵심이다. 해외에는 PG가 카드 네트워크(Visa, Mastercard)에 직접 연결되지만, 한국에서는 PG → VAN → 카드사 구조가 일반적이다.

    주요 VAN사: NICE정보통신, KIS정보통신, KSNET, 스마트로, 코밴 등

    실제 예시: 고객이 NHN KCP를 통해 카드 결제를 하면, KCP는 VAN사(예: NICE정보통신)를 경유하여 카드사에 승인 요청 전문을 보낸다. VAN사는 전문을 중계하고 건당 수수료(약 15~25원)를 받는다.

    graph LR
    A[가맹점] --> B[PG사]
    B --> C[VAN사]
    C --> D[카드사]
    D --> C
    C --> B
    B --> A

    VAN의 미래

    최근 PG사들이 VAN을 거치지 않고 카드사와 직접 연결(직접 매입)하는 추세가 늘고 있다. 토스페이먼츠는 자체 VAN 라이선스를 보유하여 VAN 중계 수수료를 절감하고 있다.

    간편결제¶

    정의: 카드·계좌 정보를 사전에 등록한 뒤, 결제 시에는 비밀번호·생체인증·PIN 등 간소화된 인증만으로 결제를 완료하는 방식이다.

    왜 중요한가: 결제 전환율을 크게 높인다. 기존 카드 인증결제 대비 결제 단계가 줄어 이탈률이 감소하며, 모바일 환경에 최적화되어 있다. 2024년 기준 한국 온라인 결제의 약 60% 이상이 간편결제를 통해 이루어진다.

    주요 서비스:

    간편결제 운영사 특징
    토스페이 비바리퍼블리카 토스 앱 기반, 높은 사용자 수
    네이버페이 네이버파이낸셜 네이버 생태계 연동, 포인트 적립
    카카오페이 카카오페이 카카오톡 기반, 송금·결제 통합
    삼성페이 삼성전자 MST+NFC 오프라인 결제, 온라인 확장
    페이코 NHN 포인트 기반 혜택, 다양한 제휴

    실제 예시: 쇼핑몰에서 결제 시 "네이버페이로 결제" 버튼을 누르면, 네이버 인증(비밀번호 또는 생체인증) 한 단계만으로 결제가 완료된다. PG사는 네이버페이의 결제 결과를 수신하여 가맹점에 전달한다.

    PCI DSS¶

    정의: Payment Card Industry Data Security Standard — Visa, Mastercard, AMEX, JCB, Discover가 공동으로 제정한 카드 데이터 보안 국제 표준이다. 카드 정보를 저장·처리·전송하는 모든 조직이 준수해야 한다.

    왜 중요한가: PCI DSS를 준수하지 않으면 카드사와 계약이 불가능하거나 벌금이 부과된다. 가맹점이 PG를 사용하는 주요 이유 중 하나가 PCI DSS 부담을 PG사에 위임하기 위해서다.

    핵심 요구사항 (12가지 중 주요 항목):

    1. 방화벽 설치·유지
    2. 카드 데이터 암호화 저장
    3. 카드 데이터 전송 시 암호화
    4. 접근 권한 최소화
    5. 정기적 보안 테스트
    6. 보안 정책 유지

    인증 레벨: 연간 카드 거래 건수에 따라 Level 1(600만 건 이상)~Level 4로 나뉘며, Level 1은 외부 감사(QSA)가 필수다.

    실제 예시: 스타트업이 자체적으로 카드번호를 저장하려면 PCI DSS Level 1 인증이 필요하고, 이는 수억 원의 비용과 수개월의 시간이 든다. 대신 Stripe나 Toss Payments를 사용하면 카드 정보가 PG 서버에만 저장되므로 가맹점의 PCI DSS 범위가 대폭 줄어든다.

    3D Secure¶

    정의: 온라인 카드 결제 시 카드사(발급사)가 추가 본인인증을 수행하는 프로토콜이다. Visa의 "Verified by Visa", Mastercard의 "SecureCode"가 대표적이다. 현재 3DS 2.0 버전이 표준이다.

    왜 중요한가:

    • 부정결제 방지: 카드 도용 시에도 발급사의 추가 인증으로 차단 가능
    • 책임 전환(Liability Shift): 3DS 인증을 거치면 부정결제 발생 시 책임이 가맹점에서 발급사로 전환됨
    • EU 규제 필수: 유럽의 PSD2(결제서비스지침) 강화인증(SCA) 요건을 충족하려면 3DS 적용이 필수

    3DS 1.0 vs 3DS 2.0:

    항목 3DS 1.0 3DS 2.0
    사용자 경험 별도 팝업 리다이렉트 인앱/인라인 인증
    인증 방식 비밀번호 고정 리스크 기반(생체, OTP 등)
    모바일 지원 미흡 최적화
    데이터 활용 제한적 100개 이상 데이터 포인트 분석

    실제 예시: 해외 쇼핑몰에서 Stripe으로 결제하면, 리스크 수준에 따라 3DS 2.0 챌린지가 발동된다. 저위험 거래는 Frictionless Flow로 추가 인증 없이 통과하고, 고위험 거래만 SMS OTP나 앱 인증을 요구한다.
